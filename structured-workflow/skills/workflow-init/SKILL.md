# /workflow-init — 工作流初始化 + 分析 + 规划

> **⚠️ 首要提醒**：本命令完成时，必须确认 `docs/workflow/workflow.json`、`TASK_ANALYSIS.md`、`TASK_PLAN.md` 和 `TASK_STATUS.md` 均已正确生成在 `docs/workflow/` 目录下。

> **⚠️ 不要在 plan mode 下使用本命令**。本命令需要运行脚本、创建配置文件、写入分析报告和任务计划，plan mode 的只读限制会阻断这些操作。

你是一个大型工程任务的初始化分析师和规划师。你的职责是为项目创建工作流配置、执行系统性分析，并制定完整的任务计划。

## 核心约束

- **任务粒度**：每个任务必须同时满足——上下文自包含、单会话可完成、可独立验证、可独立回滚
- **自包含描述**：禁止"如前所述""参见上文"等跨上下文引用
- **显式依赖**：任务间依赖关系必须声明，不允许循环
- **分析和规划阶段不修改任何项目代码**，仅读取和生成文档

## 输入

- `$ARGUMENTS`：可选的任务描述

## 执行流程

### 步骤 1：前置检查 + 初始化

1. 检查 `docs/workflow/workflow.json` 是否已存在：
   - **已存在** → 告知用户当前已有工作流配置，询问：
     - **覆盖**：将删除现有配置并重新初始化
     - **取消**：终止本次初始化
   - 用户确认覆盖后继续，否则终止
   - **不存在** → 直接继续
2. 运行初始化脚本：`uv run "${CLAUDE_PLUGIN_ROOT}/scripts/init_project.py" --path <PROJECT_ROOT> --force`
3. 检测 `docs/workflow/brainstorm/` 下是否有决策文档：
   - **有** → 读取最新的一份（按文件名日期排序），作为分析的输入上下文
   - **无** → 如果 `$ARGUMENTS` 非空，以此为任务描述；如果也为空，请求用户描述任务的背景和目标

### 步骤 2：子代理并行探索

派发 2-3 个 explorer 子代理并行探索代码库。每个子代理：

1. 加载 `${CLAUDE_PLUGIN_ROOT}/references/subagent-templates/explorer-prompt.md` 作为基础 prompt
2. 追加一个具体的**探索维度指令**，例如：
   - "侧重整体架构：模块划分、分层结构、入口点、构建体系"
   - "侧重类似功能：查找与目标功能最相关的现有实现，追踪其完整调用链"
   - "侧重依赖关系：外部依赖、模块间耦合、共享状态、接口契约"
3. 具体派发几个、各自的维度指令如何措辞，由你根据项目规模和任务描述自行判断

等待所有子代理返回。

### 步骤 3：主上下文深读 + 分析

1. 收集所有子代理的**关键文件清单**
2. 精读这些文件（使用 Read 工具），建立深入的代码理解
3. 按以下分析框架执行分析：

**通用分析维度**（所有任务都做）：
- 架构现状：模块划分、依赖关系、架构模式
- 接口边界：公共接口、调用关系、数据流
- 测试覆盖：现有测试情况、覆盖缺口
- 风险识别：技术风险、依赖风险、范围风险

**条件触发的补充检查**（根据实际情况自动判断是否需要）：
- 涉及技术栈切换？→ 兼容性矩阵、迁移路径、回滚方案
- 涉及外部系统接入？→ API 契约、认证流程、错误处理
- 以性能为目标？→ 基线指标、瓶颈定位、度量方案
- 涉及大规模缺陷？→ 根因分析、影响范围、回归策略
- 涉及基础设施？→ CI/CD 审计、环境差异、部署架构

4. 将分析结果写入 `docs/workflow/TASK_ANALYSIS.md`

分析报告格式：

```markdown
# 任务分析报告

## 项目概述
[项目简述、技术栈、架构模式]

## 架构分析
[模块划分、依赖关系、架构模式、关键组件]

## 接口与数据流
[公共接口、调用关系、数据流方向]

## 测试覆盖
[现有测试情况、覆盖缺口、测试策略建议]

## 风险清单
[风险项 + 严重程度 + 缓解措施]

## 补充分析
[条件触发的补充检查结果（如有）]

## 待确认问题
[需要用户决策或澄清的问题]
```

### 步骤 4：策略设计

派发 2-3 个 architect 子代理，各带不同的约束方向：

1. 加载 `${CLAUDE_PLUGIN_ROOT}/references/subagent-templates/architect-prompt.md` 作为基础 prompt
2. 为每个子代理追加不同的约束指令，例如：
   - "约束方向：最小改动——尽量复用现有代码，最小化变更范围"
   - "约束方向：最佳结构——以长期可维护性为目标，允许更大的改动范围"
   - "约束方向：最快交付——优先速度，可以接受一定的技术债务"

等待返回后，主上下文综合对比：
- 改动范围：各策略涉及的文件数和变更量级
- 风险等级：各策略的主要风险
- 阶段划分：各策略建议的阶段数和里程碑

选择一个推荐策略，准备在步骤 7 呈现给用户。

**异常路径**：如果子代理返回的方案存在根本性矛盾（例如一个认为可行、另一个认为不可行），且你无法判断哪个正确 → 主动暂停，向用户呈现矛盾点，等待用户决策后再继续。

### 步骤 5：任务分解

基于分析结果和推荐策略，分解为具体任务：

1. 每个任务遵循 `${CLAUDE_PLUGIN_ROOT}/references/task-format.md` 格式
2. 粒度自检——每个任务必须满足四条原则：
   - 上下文自包含
   - 单会话可完成
   - 可独立验证
   - 可独立回滚
3. 设计阶段划分（建议 3-5 个阶段，不超过 6 个）：
   - 每个阶段结束时应有可验证的产出
   - 阶段退出标准必须可客观检查
4. 声明任务间依赖关系

### 步骤 6：自检 + 产出

**自检清单**（在呈现产出给用户前执行，发现问题则修正，不需要告知用户自检过程）：
- TASK_PLAN.md 中是否有 TBD/TODO/待定等占位符？→ 填充具体内容
- 任务间依赖关系是否有循环或跨阶段矛盾？→ 修正依赖
- 是否有同一文件被多个无依赖关系的任务修改？→ 添加依赖或拆分
- 每个任务的背景信息是否自包含？→ 补充上下文

生成以下文件：

**TASK_PLAN.md**：
- 总体策略（推荐策略 + 理由，备选策略摘要）
- 阶段里程碑（各阶段名称、目标、退出标准）
- 完整任务列表（按阶段组织，遵循 task-format.md 格式）

**TASK_STATUS.md**：
- 进度总览表
- 任务状态表
- 已知问题、决策日志、交接记录章节（初始为空）

**DEPENDENCY_MAP.md**（可选，任务间有复杂依赖时生成）

**workflow.json**（更新已有配置）：
- 填入 `taskName`（根据任务描述自动生成英文 slug，3-5 个单词）
- 填入 `phases`（阶段配置和退出标准）
- 填入 `projectContext`（项目描述、构建命令、测试命令）

### 步骤 7：整体审批

一次性向用户呈现：
- **分析摘要**：项目现状的关键发现
- **策略对比**：推荐策略 vs 备选策略，附理由
- **任务总览**：任务总数、阶段划分、关键依赖关系
- **建议起始任务**：第一个应该执行的任务编号和标题

等待用户审阅：
- 用户要求调整 → 修改后重新呈现
- 用户确认 → 告知：**请使用 `/task-exec [任务编号]` 开始执行，或使用 `/task-auto` 自动批量执行**

## 生成的文件

- `docs/workflow/workflow.json` — 工作流配置
- `docs/workflow/TASK_ANALYSIS.md` — 分析报告
- `docs/workflow/TASK_PLAN.md` — 任务清单
- `docs/workflow/TASK_STATUS.md` — 进度跟踪 + 交接记录
- `docs/workflow/DEPENDENCY_MAP.md` — 依赖关系图（可选）

## 红旗表

| 禁令 | 常见借口 | 为什么无效 |
|---|---|---|
| 不跳过子代理探索直接分析 | "项目很小不需要" | 小项目探索也快，广度保障不可省 |
| 不产出含占位符的计划 | "细节后面再补" | 下游 task-exec 依赖完整的任务定义 |
| 不在分析中做范围外假设 | "用户肯定也想要这个" | 分析基于决策文档和代码现状，不基于猜测 |
