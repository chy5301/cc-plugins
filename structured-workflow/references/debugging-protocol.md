# 四阶段调试协议

本文档定义了执行过程中遇到技术故障时的系统性调试流程。task-exec 第 3 步遇到构建失败、测试报错、运行时异常时引用此文档。

---

## 四个阶段

### Phase 1：根因调查

- 读取**完整**错误信息（不只看第一行）
- 确认重现条件（每次都失败？还是偶发？）
- 检查最近的变更（`git diff`）是否引入了问题
- 检查是否有环境因素（依赖版本、配置差异）
- **收集数据，不做假设**

### Phase 2：模式分析

- 项目中有没有类似的**工作代码**可以对标？
- 错误模式指向哪个层级？（编译错误 / 运行时异常 / 逻辑错误）
- 是单点问题还是系统性问题？（一个文件报错 vs 多处报错）
- 错误信息中的关键词是否指向特定组件？

### Phase 3：假设验证

- 基于 Phase 1-2 的证据提出**一个**假设
- 设计最小测试验证该假设（只改变一个变量）
- 执行测试，记录结果
- 假设被推翻 → 回到 Phase 1 重新收集数据，不要在同一方向反复尝试

### Phase 4：实施修复

- 基于已验证的假设实施**最小修复**
- 修复后走验证门函数（`references/verification-gate.md`）确认
- 记录修复内容和根因到交接记录

---

## 升级规则

- **≥3 次修复失败**：停止调试。当前假设体系可能根本错误，需要用户介入。升级为异常 2（计划有误，停止等待确认）。
- **调试中发现是计划层面问题**（任务描述与代码实际状态不符、方案本身不可行）：立即升级为异常 2，不继续调试。

---

## 禁止行为

- 不跳过 Phase 1 直接猜测修复方向
- 不在同一假设上反复重试（失败就换假设）
- 不同时修改多处（一次只改一个变量）
- 不在没有证据时说"应该是这个问题"
