# Structured-Workflow v2 改版设计文档

## 背景与动机

structured-workflow 当前版本存在几个设计层面的改进空间：

1. **缺少需求探索环节**：用户需求模糊时，workflow-init 直接进入分析规划，容易在错误方向上做精细拆分
2. **类型分诊仪式感过重**：8 种分类 + 置信度 + 追问的流程，实际价值低于交互成本
3. **粒度约束度量维度不当**：maxFilesPerTask / maxHoursPerTask 是代理指标，不直接反映任务可执行性
4. **分析阶段依赖单一上下文**：init 自己做 Glob/Grep/Read，缺少并行广度探索
5. **验证环节过于粗略**：task-exec 的验证步骤只有两句话，缺少结构化验证协议
6. **执行中缺少调试指引**：遇到技术故障时没有系统性调试流程
7. **reference 文件组织不够合理**：部分文件只被单一 skill 使用且每次必读，应该内联

本次改版借鉴了 superpowers 插件（铁律式约束、验证门函数、系统性调试）和 feature-dev 插件（子代理并行探索、多方案架构设计、置信度过滤审查）的设计思想，结合 structured-workflow 自身的跨会话持久化优势进行整合。

## 模块设计

### 1. workflow-brainstorm

**定位**：轻量需求探索工具。只做需求澄清和方案决策，不做代码分析和任务拆分。

**流程**：

```
步骤 1：上下文感知
  - 检测 docs/workflow/ 是否存在
  - 有活跃工作流 → 读 workflow.json + TASK_STATUS.md
  - 无工作流 → 记录为 init 前探索模式

步骤 2：逐问澄清
  - 一次只问一个问题，优先选择题
  - 聚焦：目的、约束、成功标准、边界

步骤 3：方案对比
  - 提出 2-3 个方案，各附利弊
  - 给出推荐和理由，等用户选择

步骤 4：输出决策文档
  - 写入 docs/workflow/brainstorm/<日期>-<slug>.md
  - 目录不存在则自动创建

步骤 5：建议下一步
  - 无活跃工作流 → 推荐 /workflow-init
  - 有活跃工作流 → 推荐 /plan-adjust 或 /task-auto 或 /task-exec
  - 无活跃工作流时不推荐 plan-adjust / task-auto / task-exec
```

**决策文档格式**：

```markdown
## Brainstorm 结论：<标题>

### 背景与动机
<为什么要做这件事>

### 关键决策
1. <决策点 1>：选择 A（原因：...；排除 B 因为 ...）
2. <决策点 2>：...

### 约束与风险
- ...

### 建议的下一步
- [ ] 启动工作流 → `/workflow-init`
- [ ] 调整现有计划 → `/plan-adjust <具体变更>`
- [ ] 自动执行 → `/task-auto`
- [ ] 执行单个任务 → `/task-exec <任务编号>`
- [ ] 暂不行动
```

**边界**：不做代码扫描、不做任务拆分、不修改项目代码、不修改现有工作流状态文件。

**红旗表**：

| 禁令 | 常见借口 | 为什么无效 |
|---|---|---|
| 不跳过澄清直接给方案 | "需求已经很清楚了" | 你认为清楚不代表没有隐含假设 |
| 不在 brainstorm 中写代码 | "就改一行配置而已" | brainstorm 的产出是决策文档 |
| 不替用户做选择 | "方案 A 明显更好" | 呈现利弊和推荐，选择权在用户 |

### 2. workflow-init（重写）

**核心变化**：去掉类型分诊、引入子代理并行探索、内联重构后的分析/规划框架、末尾整体审批。

**流程**：

```
步骤 1：前置检查 + 初始化
  - workflow.json 已存在 → 询问覆盖/取消
  - 运行 init_project.py
  - 检测 brainstorm 目录，有则读取最新决策文档，无则请求用户描述

步骤 2：子代理并行探索
  - 派发 2-3 个 explorer 子代理（references/subagent-templates/explorer-prompt.md）
  - 各侧重不同维度（架构层次 / 类似功能 / 依赖关系）
  - 返回发现摘要 + 关键文件清单

步骤 3：主上下文深读 + 分析
  - 精读子代理返回的关键文件
  - 按通用分析框架 + 条件触发的补充检查执行分析
  - 产出 TASK_ANALYSIS.md

步骤 4：策略设计
  - 派发 2-3 个 architect 子代理（references/subagent-templates/architect-prompt.md）
  - 各带不同约束（最小改动 / 最佳结构 / 最快交付）
  - 主上下文综合对比，选择推荐策略

步骤 5：任务分解
  - 基于分析和策略分解为具体任务
  - 遵循 references/task-format.md 格式
  - 粒度自检（四条原则）
  - 设计阶段划分和退出标准

步骤 6：自检 + 产出
  - 内置自检清单：
    - 占位符扫描（TBD/TODO/待定）
    - 依赖一致性（无循环、无跨阶段矛盾）
    - 文件冲突（同一文件被多个无依赖任务修改）
    - 自包含性（无跨任务上下文引用）
  - 发现问题则修正
  - 生成 TASK_PLAN.md、TASK_STATUS.md、DEPENDENCY_MAP.md（可选）
  - 更新 workflow.json

步骤 7：整体审批
  - 呈现：分析摘要 + 策略对比 + 任务总数和阶段划分 + 建议起始任务
  - 用户要求调整 → 修改后重新呈现
  - 用户确认 → 引导 /task-exec 或 /task-auto

异常路径：任何阶段遇到高不确定性 → 主动暂停询问用户
```

**内联的分析框架**（替代 analyzer-prompts.md）：

```
通用分析维度（所有任务都做）：
  - 架构现状：模块划分、依赖关系、架构模式
  - 接口边界：公共接口、调用关系、数据流
  - 测试覆盖：现有测试情况、覆盖缺口
  - 风险识别：技术风险、依赖风险、范围风险

条件触发的补充检查（根据实际情况自动判断）：
  - 涉及技术栈切换 → 兼容性矩阵、迁移路径、回滚方案
  - 涉及外部系统接入 → API 契约、认证流程、错误处理
  - 以性能为目标 → 基线指标、瓶颈定位、度量方案
  - 涉及大规模缺陷 → 根因分析、影响范围、回归策略
  - 涉及基础设施 → CI/CD 审计、环境差异、部署架构
```

**内联的规划框架**（替代 planner-prompts.md）：

```
策略由 architect 子代理生成，主上下文从三个维度对比：
  - 改动范围：最小改动 vs 最佳结构 vs 最快交付
  - 风险等级：每个策略的主要风险
  - 阶段划分：每个策略建议的阶段数和里程碑

阶段设计原则：
  - 每个阶段结束时应有可验证的产出
  - 阶段退出标准必须可客观检查
  - 建议 3-5 个阶段，不超过 6 个
```

**红旗表**：

| 禁令 | 常见借口 | 为什么无效 |
|---|---|---|
| 不跳过子代理探索 | "项目很小不需要" | 小项目探索也快，广度保障不可省 |
| 不产出含占位符的计划 | "细节后面再补" | 下游 task-exec 依赖完整任务定义 |
| 不在分析中做范围外假设 | "用户肯定也想要这个" | 分析基于决策文档和代码现状 |

### 3. task-exec（增强）

**核心变化**：内联交接记录模板、第 4 步引用 verification-gate、异常处理新增调试分支引用 debugging-protocol、红旗表替代禁止列表。

**流程调整**：

- 步骤 0：去掉粒度约束加载，其余不变
- 步骤 3 异常处理：新增第 5 种"实施中遇到技术故障 → 引用 debugging-protocol.md"；调试后发现计划问题 → 升级为异常 2；≥3 次修复失败 → 强制升级为异常 2
- 步骤 4：引用 `references/verification-gate.md`，按 IDENTIFY→RUN→READ→VERIFY→CLAIM 执行
- 步骤 5：交接记录模板直接内联

**红旗表**：

| 禁令 | 常见借口 | 为什么无效 |
|---|---|---|
| 不做范围外变更 | "顺手改了更好" | 范围外修改可能破坏后续任务前提 |
| 不跳过验证 | "改动太小" | 验证门函数就几条命令的事 |
| 不在状态更新前汇报 | "先告诉用户好消息" | 会话中断则未持久化进度丢失 |
| 不盲目重试修复 | "再试一次应该可以" | ≥3 次失败说明假设有误 |
| 不猜测修复方向 | "应该是这个问题" | 调试协议要求先收集证据 |

**核心变化**：变更审计引入置信度过滤、构建验证引用 verification-gate、红旗表。

**流程调整**：

- 步骤 2（变更审计）：每项发现先标置信度 0-100，≥80 才进入 🔴/���/🔵 分类，<80 丢弃

**红旗表**：

| 禁令 | 常见借口 | 为什么无效 |
|---|---|---|

### 7. 粒度原则（替代 maxFilesPerTask / maxHoursPerTask）

内联在 workflow-init 和 plan-adjust 中，作为任务分解和变更时的自检标准：

1. **上下文自包含**：全新会话读完任务描述后能理解要做什么
2. **单会话可完成**：不需要"下次继续"
3. **可独立验证**：有明确验收标准，验证不依赖后续任务
4. **可独立回滚**：能在不影响其他已完成任务的前提下撤销
