# Agent-Native 设计原则

> 本文档是 Agent-Native 设计指南 Skill 的 reference 文件，规定**原则与理由**。
> 协议规约（信封形状、error 必选键、"退出码必须语义化"原则）与"规范 vs 示例"边界见 SKILL.md `## 参考文件`。退出码的具体码值是可扩展的基线约定。

## 引言

本文档定义了一套面向 Agent-Native 软件设计的原则体系。当软件的主要用户从人类变成 Agent，设计逻辑发生根本性变化：Agent 不看像素、不点按钮，它通过文本理解世界、通过工具操作世界。

本原则体系整合了三套来源：
1. **实践提炼**：从 Reader、Anecdote 等真实 Agent-Native 产品中提炼的五原则（Every.to，Dan Shipper & Claude）
2. **工程经验**：Anthropic 内部数百个 Skill 的九条写作原则（Thariq，2026）
3. **第一性原理**：从 Agent 交互特性推导的七条设计原则 + Sam Keen 的语义优先架构理论

原则分两层：**核心原则**回答"为什么这样设计"，**实践原则**回答"具体怎么做"。核心原则附带边界条件——它们不是教条，而是有适用范围的设计决策。

## 核心原则

### C1. 文本即界面

**定义**：Agent 通过文本理解能力、发现工具、处理结果。结构化文本（JSON、Markdown、CLI 输出）是 Agent 的原生交互媒介，等同于 GUI 之于人类。

**为什么重要**：
- **效率差距巨大**：CLI/工具调用方式每次交互约消耗 200 token，而 GUI Agent 需要截屏+坐标识别，token 消耗高出一个数量级。在 OSWorld 基准测试中，最佳 GUI Agent 仅达到人类水平的 47.7%，而 CLI 方式接近人类效率。
- **输出设计决定 Agent 效率**：一线开发者实践表明，结构化输出格式优化可减少 Agent 解析时间 5-9 倍——输出设计的影响往往超过工具功能本身。
- **文档是产品本身**：API 文档、权限说明、调用示例不是附属品。Agent 通过读文档来"学会"使用能力。SKILL.md 的 description 字段是给模型看的触发器，不是给人类看的营销文案。

**正面示例**：
- 所有命令支持 `--json` 标志，输出统一的 JSON 信封结构 `{success, data, metadata, error}`
- SKILL.md description 写"Monitors a PR until it merges. Trigger on 'babysit', 'watch CI', 'make sure this lands'"，而非"A comprehensive tool for monitoring pull request status"
- `--help` 输出包含参数类型、默认值、约束范围，Agent 可零解析成本理解

**边界条件——何时不完全适用**：
- **视觉任务和遗留系统**：GUI 不会消亡。它会分化为两个角色：Agent-to-Legacy-Software 的桥接层和 Agent-to-Human 的展示层（报告、仪表板）。纯文本界面无法覆盖拖拽排序、视觉布局等交互。但注意：GUI 桥接工具正在快速扩展 CLI 的覆盖面——CLI-Anything（源码路线）覆盖有源码的桌面软件，OpenCLI（浏览器路线）覆盖网站和 Electron 应用，两者合在一起已接近"一切软件"的 CLI 化。"文本即界面不适用"的范围在持续缩小。
- **人类监督场景**：企业部署中，75% 将安全、合规和可审计性列为最关键要求。监督 Agent 的人类需要可视化界面。Agent-Native 不等于"没有 GUI"，而是"Agent 不需要 GUI"。
- **消费者产品**：面向非技术终端用户的产品，GUI 仍是主要交互方式。CLI+Skill 模式最适合开发者工具和 Agent-to-Agent 场景。

### C2. 原子化与可组合

**定义**：工具以原子化基本操作暴露，功能由 Agent 在循环中组合原子操作达成。新能力通过编写新 prompt 产生，而非编写新代码。

**为什么重要**：
- **涌现能力是核心价值**：当工具足够原子化，Agent 能完成开发者没有明确设计过的任务。例如"把会议记录和任务列表交叉对比，告诉我哪些已承诺但未排期的事项"——通过 `list_files`、`read_file`、`search` 的组合涌现。
- **可组合性意味着零代码扩展**：有了原子工具和对等性，只需编写新 prompt 就能创建新功能。用户甚至可以自己修改 prompt 定制行为。一个"周回顾"功能不需要写代码实现——写一段 prompt 描述期望结果，Agent 用现有工具自行编排。
- **检验标准**：要改变行为时，你是在修改 prompt 还是重构代码？如果是后者，说明粒度不够细。

**正面示例**：
- 提供 `read_file`、`write_file`、`move_file` 三个原子工具，而非 `classify_and_organize_files()` 一个复合工具
- Skill 设计清晰地落在一个类别中（库参考、数据查询、流程自动化等），而非横跨多个
- CLI-Anything 项目的后端包装模式：`utils/<software>_backend.py` 暴露原子操作，Agent 自行编排

**边界条件——何时不完全适用**：
- **可靠性门槛**：Agent 每步准确率约 85% 时，10 步工作流成功率仅约 20%（0.85^10）。过度原子化意味着更多步骤、更多失败点。在可靠性未达到 99%+ 的当前阶段，关键路径上适当合并操作（减少步骤数）可能比极致原子化更务实。
- **Token 与延迟成本**：每次工具调用都有 token 开销和延迟。单次查询调用模型 1 次，Agent 工作流可能调用 50-500 次。过度原子化在规模化场景下成本倍增。
- **复杂度预算**：三行相似代码优于一个过早的抽象。原子化服务于可组合性，而非服务于自身。当工具只在单一场景使用、不存在组合需求时，合并为更粗粒度的操作更合理。

### C3. 对等性

**定义**：用户通过 UI 能做的一切，Agent 必须能通过工具同等实现。对等性是基础原则——没有它，其他原则都失去意义。

**为什么重要**：
- **Agent 只能访问部分功能就无法替代用户完成完整任务**。笔记应用中，用户能创建、打标签、搜索、删除笔记，Agent 也必须有对应的工具或工具组合。
- **检验方法**：随意选一个 UI 操作，Agent 能通过你提供的工具完成吗？如果不能，说明对等性不足。
- **"先构建再暴露"是反模式**：传统方式开发完整产品后再接入 Agent，往往只暴露部分功能。Agent-Native 设计从一开始就保证工具层和 UI 层的能力对等。

**正面示例**：
- 笔记应用提供完整的 CRUD + 搜索 + 标签管理工具集，覆盖 UI 所有操作
- CLI-Anything 将 Blender、GIMP、LibreOffice 等 GUI 软件改造为 CLI，核心原则是"Use the Real Software — Don't Reimplement It"
- LibTV 案例的 Skill 即薄中继模式：Skill 不编排工作流，只转发用户意图给后端 Agent，确保所有能力都可通过 Skill 访问

**边界条件——何时不完全适用**：
- **有意的不对等是安全设计**：企业场景中，HITL（Human-in-the-Loop）分层是标准实践。Level 1（Agent 建议，人类决定）到 Level 4（完全自主）的层级设计意味着某些操作被有意限制给人类执行。60% 的企业限制 Agent 在无人类监督下访问敏感数据——这不是设计缺陷，而是安全决策。
- **UI 原生交互难以映射**：拖拽排序、可视化布局、手势操作等 UI 交互很难映射为工具操作。对这类交互，接受"功能近似"而非"完全对等"是务实选择。
- **渐进式实现**：对等性是方向而非前提条件。P0 阶段覆盖核心操作的 `--json` + 输入验证即可，不必一次性映射所有 UI 操作。

### C4. 安全边界优先

**定义**：权限声明式、最小化、可审计。先定义边界，在边界内自由操作——比逐次确认更实用。

**为什么重要**：
- **Agent 安全的"原罪"**：Agent 需要读取外部数据才能工作，但外部数据可能包含恶意指令。OWASP Agentic Security Top 10 将 Prompt Injection（ASI01）列为头号威胁——35% 的真实 AI 安全事件仅由简单提示触发。
- **生态现状堪忧**：MCP 生态在 60 天内累计 30+ CVE，492 个服务器暴露且零认证，36.7% 存在 SSRF 漏洞。88% 的组织已报告 AI 安全事件。
- **"定义边界后在边界内自由操作"优于"逐次确认"**：Claude Code 的沙箱实践表明，预定义权限边界减少了 84% 的权限提示，同时保持了安全性。
- **可逆性是安全边界的延伸**：危险操作必须可撤销或至少要求显式确认。Agent 会犯错，系统应该容错。`--dry-run` 和确认机制是安全边界的具体实现。

**正面示例**：
- 工具声明所需权限级别（只读/读写/系统级），平台据此授权
- 分级授权模型：Level 0（只读）→ Level 1（本地写入）→ Level 2（网络访问）→ Level 3（系统操作）→ Level 4（不可逆操作需人工确认）
- Anthropic 的 `/careful` Hook：碰生产环境时才激活，拦截 `rm -rf`、`DROP TABLE`、`force-push` 等危险操作——按需激活而非全局永远开启
- 输入参数严格验证：JSON Schema 校验、路径安全检查（防止 `../../etc/passwd` 穿越）

**反面示例——OpenCLI 的全权委托模型**：
- OpenCLI 通过浏览器将网站 CLI 化，Agent 默认获得用户在浏览器中的全部权限（发消息、提交表单、社交操作），无分级、无确认、无沙箱
- 本地 Daemon 监听 localhost:19825 且无身份验证——任何本机进程都可冒充用户操作浏览器
- 这是"凭证不离开浏览器 ≠ 安全"的典型案例：关键不是凭证在哪，而是 Agent 能用它做什么

**边界条件——何时不完全适用**：
- **安全疲劳**：过度的安全确认会降低采用率。如果每个操作都弹确认框，Agent 的自主性优势荡然无存。安全设计的目标是"恰当的安全"，不是"最大的安全"。
- **个人开发工具场景**：个人使用的本地工具，安全模型可以大幅简化。五级分层授权对个人 CLI 工具是过度工程。
- **安全与速度的权衡**：企业在 Agent 安全治理上投入数千万美元，但 Gartner 警告超 40% 的 Agent 项目将被取消——过度治理是项目失败的原因之一。安全设计应匹配场景的实际风险等级。

## 实践原则

### P1. 可发现性

**定义**：Agent 应能自主发现工具的能力，无需预先编程。`--help`、schema 描述、能力清单是 Agent 时代的"首页"。

**为什么重要**：Agent 面对一个陌生工具时，第一步是理解它能做什么。如果 `--help` 输出的信息不足以让 Agent 判断参数类型、约束范围和预期行为，Agent 要么猜测（导致错误），要么放弃（功能浪费）。

**正面示例**：
- `--help` 输出包含每个参数的类型、默认值、约束范围和简短说明
- 支持 `--help --json` 输出机器可读的命令 schema
- SKILL.md 的 description 字段包含触发短语，让 Agent 知道何时该调用
- 首次使用时，如果必要配置缺失，主动引导设置（而非静默失败）

---

### P2. 确定性

**定义**：相同输入产生相同输出，状态变更可预测。Agent 的决策链依赖确定性，模糊的返回值会让整条链路崩溃。

**为什么重要**：Agent 在循环中工作——读取输出、判断下一步、执行、再读取。如果输出格式不稳定或语义模糊，Agent 的判断会漂移。工具提供"硬确定性"（相同输入相同输出），Prompt 提供"软引导"（偏好和风格）。推荐的设计位置是 Sam Keen 确定性光谱中的"Prompt 引导型"——工具约束中等、Prompt 引导高，平衡操作清晰度与领域灵活性。

**正面示例**：
- 成功和失败返回统一的 JSON 信封结构，Agent 只需检查 `success` 字段
- 错误输出包含错误码（机器可读）+ 错误描述（人类可读）+ 建议操作
- 幂等操作明确标记，Agent 可安全重试
- 支持 `--no-interactive` 模式：禁用交互式提示、pager 和确认对话框。反面教训：AWS CLI v2（2019）将默认 pager 改为 `less`，导致全球数千个 CI 管道挂起——Agent 无法回答 `Are you sure? [y/N]`

---

### P3. 输出即产品

**定义**：输出格式设计比功能设计更影响 Agent 效率。砍掉冗余输出比添加新功能更能提升 Agent 体验。

**为什么重要**：一线开发者实践表明，结构化输出优化可减少 Agent 解析时间 5-9 倍。MCP 生态中，工具描述的 token 膨胀是主要痛点——每次调用都要把完整 schema 塞进 context。CLI 的 token 效率比 MCP 高约 33%，核心原因是输出更精简。

**正面示例**：
- 默认输出面向人类可读，`--json` 输出面向 Agent——两种模式共存
- JSON 输出只包含 Agent 需要的字段，不重复 metadata
- 支持 `--fields` 参数让 Agent 选择需要的字段，减少输出体积
- 遵循 Anthropic 的"别说废话"原则——Skill 只写能把 Agent 推出默认思维模式的信息

---

### P4. Gotchas 驱动迭代

**定义**：Skill 的核心价值不在 Day 1 的初始文档，而在持续积累的踩坑记录。信息密度最高的部分永远是 Gotchas。

**为什么重要**：Anthropic 内部经验显示，billing-lib Skill 从 Day 1 的简单说明演化到 Month 3 的完整踩坑手册（"Proration rounds DOWN, not to nearest cent"、"test-mode 跳过 webhook"、"幂等键 24 小时过期"）。这些 Gotchas 才是 Skill 真正"活"起来的方式。每次 Agent 踩一次坑，就加一行。

**正面示例**：
- Skill 维护专门的 Gotchas 段落或文件，持续追加
- 跨会话记忆：通过文本日志或 SQLite 持久化使用历史，Agent 自动对比变化
- 版本迭代不需要发版——修改 Gotchas 即生效

---

### P5. 渐进式信息披露

**定义**：把文件系统当作上下文工程（context engineering）的工具。Skill 不是一个大文件，而是一个文件夹——SKILL.md 是精简的入口中枢，详细内容按需加载。

**为什么重要**：Agent 的上下文窗口是稀缺资源。一次性加载所有信息既浪费 token 又稀释关键内容。queue-debugging Skill 的设计示范了这一点：SKILL.md 仅 30 行，根据症状指向 `stuck-jobs.md`、`dead-letters.md`、`retry-storms.md`，详细函数签名放在 `references/api.md`。Agent 按需读取，不会一次加载全部。

**正面示例**：
- SKILL.md 控制在 2000 词以内，只包含决策框架和导航索引
- 详细知识拆分到 `references/` 子目录，每个文件自包含
- 提供现成函数库（`lib/signups.py`），Agent 直接 import 组合——给代码，别给指令

---

### P6. 约束意图而非步骤

**定义**：说清目标和边界，把方法交给 Agent。Skill 指令留足灵活度，避免把 Step 1 到 Step 6 都写死。

**为什么重要**：Skill 是高度复用的——不同上下文需要不同执行路径。过度约束步骤意味着 Skill 只能处理它被明确设计过的场景，丧失了 Agent 的推理优势。

**正面示例**：
- **过度约束**："Step 1: git log → Step 2: cherry-pick → Step 3: git status → Step 4: 逐个解决冲突……"
- **恰当约束**："Cherry-pick the commit onto a clean branch. Resolve conflicts preserving intent. If it can't land cleanly, explain why."
- Sam Keen 的"Prompt 引导型"设计位置：工具提供中等约束（参数类型、边界校验），Prompt 提供高引导（目标、偏好、质量标准）

## 反模式清单

以下是违反上述原则的常见设计错误：

| 反模式 | 违反原则 | 说明 |
|--------|---------|------|
| **Agent 作为路由器** | C2, C3 | Agent 只分发请求到不同后端，不参与推理决策。相当于把 Agent 降级为 API Gateway |
| **先构建再暴露** | C3 | 按传统方式开发完整产品后才考虑 Agent 接入。结果：只有部分功能可通过工具访问 |
| **工作流化的工具** | C2 | 把多步决策逻辑打包进单个工具（如 `classify_and_organize_files`）。Agent 无法介入中间步骤 |
| **请求/响应思维** | C2 | 设计为单次交互而非"循环直到完成"。Agent 天然在循环中工作 |
| **忽视输出设计** | P3 | 功能正确但输出冗余、格式不稳定。Agent 花费大量 token 解析无关信息 |
| **步骤化指令** | P6 | Skill 把操作步骤写死，Agent 无法根据上下文调整执行路径 |
| **一次性加载** | P5 | 把所有信息塞进一个文件。Agent 被迫加载不相关的上下文 |
