# Agent-Native 架构模式

> 本文档是 Agent-Native 设计指南 Skill 的 reference 文件。
> 提供三层架构定义、协议选择矩阵、复杂度分级架构方案和展示层演进路线。
> 关联原则：C1 文本即界面、C2 原子化与可组合、C3 对等性、C4 安全边界优先、P5 渐进式信息披露、P6 约束意图而非步骤（见 design-principles.md）

---

## 核心认知

Agent-Native 架构的本质是**一个内核，两个入口**——Agent 和人类通过不同的接口层访问同一份业务逻辑和数据。

| 维度 | 传统架构 | Agent-Native 架构 |
|------|---------|------------------|
| 入口 | 1 个（GUI 或 API） | 2+（Agent 控制层 + 人类展示层） |
| 业务逻辑 | 分散在 UI/API/后端 | 集中在数据层，入口层只做转发 |
| UI 角色 | 产品核心 | 翻译层 / 渲染终端 |
| 能力暴露 | 写 API 文档 | SKILL.md + `--help --json` + Schema 自省 |
| 分发方式 | App Store / Web | SKILL.md / ClawHub / Plugin |
| 扩展方式 | 写代码 | 写 Skill（prompt + 配置） |

**量化证据**：CLI token 效率比 MCP 高 33%；CLI `--help` 仅 ~200 tokens vs MCP 连接注入 ~55,000 tokens；结构化输出优化可将 Agent 解读时间降低 5-9 倍。

**经济学驱动**：产品形态从 App 变成 Skill 不仅是技术选择，也是商业模式变迁。传统 SaaS 按席位收费（毛利率 78-85%），AI-First 产品按结果收费（毛利率 55-70%，可变推理成本占 20-40%）。当 Agent 可以按需生成和组合 Skill，分发单元从"完整应用"变成"原子能力"——**可组合性比完整性更有价值**，数据和领域知识成为护城河（代码可瞬间生成，专有数据无法生成）。这一趋势解释了为什么架构应以"Skill 而非 App"为设计目标，也解释了为什么企业场景仍需要治理层和审计层——因为按结果收费意味着每次调用都有成本，成本可预测性成为架构约束。

**结论**：三层分离是 Agent-Native 架构的基础模式。先把数据层做实，再分别暴露 Agent 入口和人类入口。

## 1. 三层架构

```
┌─────────────────────────────────────────────────┐
│              控制层（Agent 入口）                  │
│   CLI + --json  │  MCP Server  │  A2A Agent Card │
├─────────────────────────────────────────────────┤
│              展示层（人类入口）                    │
│  CLI 默认输出  │  --report HTML  │  A2UI / Web   │
├─────────────────────────────────────────────────┤
│              数据层（共享内核）                    │
│  业务逻辑  │  文件系统/DB  │  后端 API/引擎       │
└─────────────────────────────────────────────────┘
```

### 1.1 控制层

**职责**：面向 Agent 的结构化接口。Agent 通过此层感知能力（可发现性）和操作系统（工具调用）。

**核心要求**：
- 所有输出必须支持 `--json` 结构化模式（原则 C1、P3）
- 输入验证前置——假设输入始终具有对抗性（原则 C4）
- 能力自描述——`--help --json` 或 `schema` 子命令（原则 P1）
- SKILL.md 提供高层语义描述（原则 P5）

**协议选择**：默认 CLI+Skill，复杂场景可叠加 MCP 表面（详见第 2 节）。

### 1.2 展示层

**职责**：面向人类的可视化和监督接口。在 Agent-Native 架构中，展示层不承载业务逻辑——它是数据层状态的"渲染终端"。

**渐进式选择**（详见第 4 节）：
- **P0**：CLI 默认人类可读输出（零额外成本）
- **P1**：`--report` 生成静态 HTML（低成本可视化）
- **P2**：Streamlit/Gradio 快速 Dashboard（需要交互时）
- **P3**：A2UI 声明式 UI（跨平台标准化渲染，等 v1.0 稳定）

**企业场景特殊需求**：HITL 审批界面、操作审计日志、权限控制面板。这些需求使展示层在企业场景中不可省略——企业需要 GUI 来监督 Agent（原则 C4）。

### 1.3 数据层

**职责**：业务逻辑的唯一事实来源。两个入口共享同一份状态和逻辑。

**设计要求**：
- **状态共享**：Agent 创建的内容在展示层可见，反之亦然（原则 C3）
- **幂等操作**：相同输入产生相同结果，支持安全重试（原则 P2）
- **后端包装**：包装真实软件的 API，不用语言重新实现功能（CLI-Anything 核心教训："Use the Real Software — Don't Reimplement It"）

**实现方式随复杂度递增**：
- 简单工具：文件系统（读写文件即完成任务）
- 中等服务：文件系统 + 轻量数据库（SQLite、JSON 存储）
- 复杂平台：后端 API + 数据库 + 后端 Agent（LibTV 模式）

**关键案例**：LibTV 的两个入口（无限画布 Web GUI + OpenClaw Skill）共享同一个 project/session 状态——Agent 通过 Skill 创建的视频内容会自动出现在画布项目中，人类可在画布上继续精调。

## 2. 协议选择

### 2.1 四协议速查

| 协议 | 最佳场景 | Token 成本 | 开发复杂度 | 平台覆盖 |
|------|---------|-----------|-----------|---------|
| CLI+Skill | 默认选择，开发者工具、系统管理 | 低 | 低 | 全部（3/3 平台） |
| MCP | 复杂结构化 API、需类型安全 | 高 | 中 | 全部（3/3 平台） |
| A2A | 多 Agent 协作（未来） | 中 | 高 | 暂无原生支持 |
| OpenAPI/REST | 已有 REST API 的桥接 | 低 | 低 | 通过 curl/MCP 包装 |

### 2.2 决策路径

```
开始
 │
 ├─ 工具是否需要 Agent 调用？
 │   └─ 否 → 不需要 Agent-Native 改造
 │   └─ 是 ↓
 │
 ├─ P0：实现 CLI + --json 输出 + 输入验证
 │   （覆盖所有 Agent 平台）
 │
 ├─ P1：编写 SKILL.md + --help --json
 │   （提升可发现性，Claude Code/OpenClaw 增强体验）
 │
 ├─ P2：添加 --dry-run + --fields
 │   （提升安全性和输出效率）
 │
 └─ P3：是否有复杂嵌套参数、shell 转义成为瓶颈？
     └─ 是 → 叠加 MCP Server 表面
     └─ 否 → CLI+Skill 已足够
```

**核心原则**："一个事实来源，两个界面"——核心逻辑实现一次，CLI 和 MCP 共享同一份 schema 和业务逻辑。MCP 是 CLI 的补充而非替代。

### 2.3 平台覆盖

| 协议 | Claude Code | OpenCode | OpenClaw |
|------|------------|----------|----------|
| CLI+Skill | ✅ 完整 | ⚠️ 仅 Bash（无 SKILL.md） | ✅ 完整 |
| MCP | ✅ 完整 | ✅ stdio | ✅ MCP Registry |
| A2A | ❌ | ❌ | ❌ |

OpenCode 不支持 SKILL.md，但可通过 `--help --json` + Bash 工具覆盖。这是 P1 中 `--help --json` 与 SKILL.md 并列的原因。

## 3. 架构选择矩阵——按复杂度分级

| 维度 | 简单 CLI 工具 | 中等复杂度服务 | 复杂平台 |
|------|-------------|--------------|---------|
| **典型场景** | 文件处理、格式转换、系统查询 | 数据查询、CI/CD、API 聚合 | 视频生成、企业业务系统、SaaS |
| **控制层** | CLI + SKILL.md | CLI + SKILL.md（+ MCP 可选） | CLI/MCP + 后端 Agent |
| **展示层** | CLI 默认输出 | `--report` HTML | Web Dashboard / A2UI |
| **数据层** | 文件系统 | 文件系统 + 轻量 DB | 后端 API + 数据库 |
| **HITL 级别** | L3-L4（高自主） | L2-L3（执行+审批） | L1-L2（人类监督） |
| **Skill 结构** | 单文件 SKILL.md | SKILL.md + references/ | SKILL.md + references/ + 后端文档 |
| **参考案例** | CLI-Anything 单工具 | standup-post、funnel-query | LibTV |

### 3.1 简单 CLI 工具

**架构**：

```
Agent/用户 → CLI（Python Click/argparse）→ 后端引擎/文件系统
```

**特征**：
- 控制层只需 CLI + `--json`，展示层靠 CLI 默认输出
- SKILL.md 即全部文档，无需 references/ 子目录
- 后端包装模式：`utils/backend.py` 调用真实软件 API，不重新实现（原则 C2）
- 命令分组遵循标准五组：Project / Core / Import-Export / Config / Session

**参考**：CLI-Anything 的 11 个 GUI→CLI 改造均采用此架构。关键教训——CLI 必须调用真实软件后端（如 GIMP 的 Script-Fu），用 Pillow 替代 GIMP 会丢失滤镜效果、色彩管理和格式兼容性。

### 3.2 中等复杂度服务

**架构**：

```
Agent/用户 → CLI → 后端 API / DB → --report HTML（可选）
```

**特征**：
- 控制层 CLI+Skill，展示层按需生成 HTML 报告
- SKILL.md 作为入口，references/ 存放详细 API 文档和 Gotchas（原则 P5：渐进式信息披露）
- 状态持久化到轻量 DB（SQLite）或日志文件
- 可选叠加 MCP 表面（当参数结构复杂时）

**参考**：Anthropic 内部 Skill 类型中的"数据查询与分析"（funnel-query：连接数据源，返回结构化结果）和"业务流程自动化"（standup-post：聚合多源数据生成报告）。

### 3.3 复杂平台

**架构**：

```
Agent → Skill（薄中继）→ 后端 Agent/API → 数据库
                                ↑
人类 → Web Dashboard / A2UI ────┘
```

**特征**：
- **Skill 即薄中继**：Skill 只做触发和通信，核心能力留在后端（"接口而非大脑"）（原则 P6）
- 展示层需要完整 Web Dashboard 或 A2UI 声明式 UI
- IM 异步会话协议：支持多轮交互、耗时较长的任务
- 两个入口共享同一后端状态

**企业级约束**（需额外考虑的 6 项设计维度）：

| 约束 | 设计要求 |
|------|---------|
| 可审计性 | 每个 Skill 调用产生可追溯日志 |
| 授权层级 | Skill 声明所需权限级别和影响范围 |
| HITL 接口 | 定义哪些操作需人类审批 |
| 优雅降级 | Agent 不可用时回退到人类可操作模式 |
| 成本可预测性 | 声明资源消耗模式（token、API 调用次数等） |
| 多 Agent 编排 | 考虑被编排在多 Agent 工作流中的场景 |

**参考**：LibTV 的双接口设计（无限画布 + OpenClaw Skill 共享同一 project 状态），以及 Bank of America Erica AI（90%+ 员工使用，HITL 贯穿全流程）。

## 4. 展示层演进路线

| 阶段 | 方案 | 成本 | 适用条件 | 关联原则 |
|------|------|------|---------|---------|
| P0 | CLI `--json` + 默认人类输出 | 零 | 所有工具的起点 | C1、P3 |
| P1 | `--report` 静态 HTML | 低 | 需要可视化但不需交互 | C3 |
| P2 | Streamlit / Gradio Dashboard | 中 | 需要实时交互式可视化 | — |
| P3 | A2UI 声明式 UI | 中→高 | 跨平台标准化渲染，A2UI v1.0 稳定后 | — |

**决策建议**：

- **默认从 P0 开始**——CLI 双模输出（`--json` 给 Agent，默认格式给人类）满足 80% 场景
- **需要可视化时进入 P1**——`--report` 生成自包含 HTML，人类用浏览器查看，成本极低
- **企业场景需要 HITL 时考虑 P2+**——Streamlit 可快速搭建审批界面和监控面板
- **A2UI 是未来方向但非当前必需**——v0.8 Public Preview，等 v1.0 稳定再评估

**Agent 入口趋同，人类入口因场景而异**。Agent 都走 CLI/MCP 的结构化路径，但人类的需求从"看个结果"到"精细操控"差异巨大，展示层需要按需演进。

## 5. 四层协议栈

完整的 Agent-Native 通信栈包含四层，但 **MVP 只需关注底部**：

```
┌──────────────────────────────────────┐
│    UI 层（React / Flutter / 终端）    │  ← 渲染
├──────────────────────────────────────┤
│  AG-UI：运行时通信（传输层）           │  ← 如何传递（事件流）
├──────────────────────────────────────┤
│  A2UI：声明式 UI 描述（内容层）        │  ← 传递什么（JSON 组件树）
├──────────────────────────────────────┤
│  MCP / CLI+Skill：工具调用（能力层）   │  ← Agent 能做什么
├──────────────────────────────────────┤
│  A2A：Agent 间协作（协作层）          │  ← Agent 间如何协作
└──────────────────────────────────────┘
```

| 层 | 成熟度 | 投入建议 |
|----|--------|---------|
| 能力层（CLI+Skill / MCP） | 高 | **立即投入**——这是 Agent-Native 的基础 |
| 内容层（A2UI v0.8） | 中 | 关注演进，等 v1.0 |
| 传输层（AG-UI） | 中 | 与 A2UI 配套，同步关注 |
| 协作层（A2A v0.3） | 低 | 暂不投入，等生态成熟 |

## 6. 反模式

| 反模式 | 违反原则 | 说明 |
|--------|---------|------|
| **两套后端** | C3 对等性 | CLI 和 Web 各连不同数据源，Agent 创建的内容人类看不到 |
| **Skill 即大脑** | C2、P6 | Skill 内编排复杂工作流，而非将意图转发给后端处理 |
| **先产品后接入** | C3 对等性 | 先构建完整 GUI 产品，再"把 API 暴露给 Agent"——通常导致能力打折 |
| **全权委托** | C4 安全边界 | Agent 获得用户全部权限，无分级授权、无操作确认 |
| **过早标准化** | — | MVP 阶段就引入 A2UI/AG-UI，增加不必要的复杂度和依赖 |
| **重新实现后端** | C2 | 用 Python 重写 GIMP 滤镜而非调用 Script-Fu——丢失精度和兼容性 |

## 附录：与设计原则的映射

| 架构概念 | 对应原则 | 关系说明 |
|----------|---------|---------|
| 三层分离 | C1、C3 | Agent 通过文本（控制层）交互，人类通过视觉（展示层）交互，共享数据层保证对等 |
| CLI+Skill 为默认协议 | C1、P3 | 文本是 Agent 的原生媒介，CLI 是 token 效率最高的接口 |
| 后端包装模式 | C2 | 包装真实软件能力，保持原子化，不重新实现 |
| 薄中继模式 | C2、P6 | Skill 约束意图（做什么），后端负责执行（怎么做） |
| 渐进式展示层 | P5 | 按需演进（CLI→HTML→Dashboard→A2UI），不过早引入复杂度 |
| HITL 分级 | C4 | 安全边界优先，有意的不对等是安全设计而非缺陷 |
| Skill 文件夹分层 | P5 | SKILL.md 为入口，references/ 按需加载，避免 context 过载 |
| Gotchas 积累 | P4 | 架构决策中的踩坑记录是最高密度的知识——持续追加 |
