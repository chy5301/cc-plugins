# 验证门函数

本文档定义了任务完成验证的标准协议。task-exec 第 4 步和 phase-review 步骤 4 引用此文档。

---

## 五步验证协议：IDENTIFY → RUN → READ → VERIFY → CLAIM

### 1. IDENTIFY（识别）

确定本次验证需要什么证据：
- 构建通过？→ 需要构建命令的完整输出
- 测试通过？→ 需要测试命令的完整输出
- 功能正确？→ 需要逐项检查验收标准
- 无回归？→ 需要完整测试套件输出

### 2. RUN（执行）

执行验证命令。不假设结果，不跳过任何命令。

### 3. READ（读取）

读取**完整输出**，不只看 exit code：
- 构建输出中是否有警告？
- 测试输出中是否有跳过的用例？
- 是否有 deprecation 提示？

### 4. VERIFY（核实）

将输出与预期逐项对照：
- 验收标准中的每一条是否都有对应的证据？
- 证据是否直接来自命令输出（不是推理）？

### 5. CLAIM（声称）

全部通过才能声称完成。任何一项缺少证据，回到步骤 2 补充验证。

---

## 常见失败理由对照表

| 声称 | 需要的证据 | 无效理由 |
|---|---|---|
| "构建通过" | 构建命令的完整输出，exit code 0 | "应该没问题""上次是通过的" |
| "测试通过" | 测试命令的完整输出，全部 pass | "只改了一点不会影响""太简单不用测" |
| "功能正常" | 验收标准逐项检查结果 | "我看了代码逻辑是对的" |
| "没有回归" | 完整测试套件输出 | "只改了这个文件不会影响别的" |
